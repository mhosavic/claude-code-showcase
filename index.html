1.Installer

Deux chemins, choisis selon ce que tu veux faire.

Option A — cloner le repo (recommandé)

git clone https://github.com/mhosavic/claude-code-showcase.git
cd claude-code-showcase
claude

Le repo embarque un .claude/settings.json qui propose les quatre plugins au premier lancement — tu n'as qu'à accepter le prompt. Avantage : tu as le code en local pour le lire dans ton éditeur, et /showcase-tour:inspect peut te promener dans n'importe quel fichier.

Option B — installer juste les plugins (sans cloner)

/plugin marketplace add mhosavic/claude-code-showcase
/plugin install draft-email@claude-code-showcase
/plugin install commit-helper@claude-code-showcase
/plugin install linkedin-post@claude-code-showcase
/plugin install showcase-tour@claude-code-showcase
/reload-plugins

À taper directement dans Claude Code, depuis n'importe quel répertoire. Plus rapide; tu n'as juste pas le code source en local pour le lire dans ton éditeur (mais /showcase-tour:inspect fonctionne quand même contre les copies en cache des plugins).

2.Lancer la visite guidée

Une fois les plugins installés, dans Claude Code : 

/showcase-tour:tour fr

Compte ~15 minutes. Elle te promène à travers les six questions, une à la fois, avec une commande exacte à essayer pour chacune. Tu peux aussi passer quick (5 min, juste l'essentiel) ou deep (30 min, on regarde le code).

3.Tes six questions, répondues

Les cartes ci-dessous sont là pour que tu puisses naviguer si tu veux sauter des sections. Chacune renvoie au code et au doc correspondant. Quand le pattern fonctionne différemment dans Cowork, c'est noté.

Q1

La structure d'un skill simple — exemple : rédiger un courriel

"La structure des fichiers pour un skill simple (exemple, rédiger un courriel)."

Trois fichiers en tout : le manifest, le SKILL.md, et le README. Un skill, c'est un prompt en markdown avec un en-tête YAML — Claude le déclenche quand tu tapes /draft-email:draft.

Dans Cowork : même structure de fichier. La distribution change : un skill atterrit dans Cowork via une soumission au catalogue à claude.com/plugins, ou en collant le corps du skill dans les instructions personnalisées d'un Project.

Code : plugins/draft-email/  ·  Doc : docs/01-simple-skill.md

Q2

Orchestration d'un skill complexe — un post LinkedIn

"Un projet qui a comme tâche de faire un post LinkedIn avec plusieurs skills qui lui sont propres : questionner l'utilisateur, générer le texte, générer l'image, pousser le brouillon."

Pile ce que tu décris. /linkedin-post:post est un skill orchestrateur qui appelle quatre sous-skills dans l'ordre : entrevue → rédaction → génération d'image → publication. Plus un sous-agent en alternative (post-coordinator) qui montre le pattern « subagent-as-orchestrator ».

Dans Cowork : l'orchestration fonctionne pareil. La seule différence c'est le serveur MCP derrière (voir Q4) — il tourne en HTTP plutôt qu'en stdio, et s'ajoute comme connecteur dans Paramètres → Connecteurs.

Code : plugins/linkedin-post/  ·  Doc : docs/02-complex-skill-orchestration.md

Q3

Partage avec l'équipe dans Claude Team

"Pour les skills simples, nous utilisons une marketplace sur GitHub. Pour les skills complexes, comment les rendre disponibles?"

Bonne nouvelle : c'est exactement le même mécanisme. Un plugin complexe (avec sous-skills, sous-agent, serveur MCP) se distribue via le même marketplace.json qu'un skill simple. La différence se trouve dans le contenu du dossier plugins/<nom>/. Trois patterns documentés : ajout manuel par chaque dev, bootstrap automatique via extraKnownMarketplaces dans le settings du projet, ou push admin via les server-managed settings.

Dans Cowork : les marketplaces GitHub personnalisés ne sont pas (encore) supportés. Les chemins qui marchent aujourd'hui — soumission au catalogue Anthropic à claude.com/plugins, instructions personnalisées d'un Project, ou push admin (en cours de déploiement). Détails dans le doc Cowork plus bas.

Doc : docs/03-team-distribution.md

Q4

Serveur MCP avec identifiants + restriction par skill

"La structure d'un serveur MCP avec clés d'accès/identifiants (j'ai oublié le terme que tu m'as dit) pour gérer quelles fonctions peuvent être appelées par quels skills."

Le terme c'est userConfig — un champ du plugin.json qui déclare les identifiants à demander à l'install. Claude Code les stocke dans le keychain de l'OS, puis les injecte comme variables d'environnement quand le serveur MCP démarre. La restriction par skill se fait via allowed-tools: dans l'en-tête de chaque skill — l'orchestrateur peut publier sur LinkedIn, le skill d'image peut générer, l'entrevue n'a aucun accès au serveur.

Dans Cowork : le serveur MCP embarque deux transports — stdio (Claude Code) et HTTP Streamable (Cowork comme connecteur perso). Les identifiants passent par variables d'environnement côté hébergement (ou OAuth en production). La restriction par outil se fait dans Paramètres → Connecteurs → ton connecteur, en activant ou désactivant chaque outil.

Code : plugins/linkedin-post/mcp-server/  ·  Doc : docs/04-mcp-server-with-auth.md

Q5

Un exemple de fonction MCP

"Un exemple de fonction MCP."

generate_image — un outil MCP complet : schéma Zod pour les arguments, mode simulation par défaut, vrai appel à OpenAI gpt-image-1 en mode réel. Le repo couvre les trois primitives MCP — outils (verbes que Claude appelle), invites (commandes slash gabarits), ressources (contenu attachable avec @).

Dans Cowork : le code de la fonction est identique — c'est le même protocole MCP. Seul le point d'entrée change (HTTP au lieu de stdio). Les outils apparaissent dans Cowork une fois le connecteur ajouté.

Code : mcp-server/src/tools/images.ts  ·  Doc : docs/05-mcp-function-example.md

Q6

Ajouter un MCP dans Claude Team comme connecteur

"Pour que nous n'aillions pas à le faire sur chaque poste si possible."

Deux chemins. (a) Serveur MCP HTTP hébergé → enregistré une fois dans la console admin Claude.ai → apparaît automatiquement chez tous les utilisateurs. (b) Serveur MCP stdio embarqué dans un plugin → distribué via le marketplace poussé par les server-managed settings. Un arbre de décision dans la doc pour choisir entre les deux.

Dans Cowork : le chemin (a) fonctionne identiquement — c'est le seul point où Claude Code et Cowork sont parfaitement convergents. Le chemin (b) est Claude Code seulement, mais on règle ça en passant à (a) pour les utilisateurs Cowork.

Doc : docs/06-claude-team-connectors.md

4.Cowork — ce qui traverse, ce qui ne traverse pas

Voici ce qui marche et ce qui ne marche pas aujourd'hui :

Ce qui traverse — le serveur MCP (les outils, les invites, les ressources). Le code est identique; seul le transport change (HTTP au lieu de stdio). Tu déploies le serveur, ton équipe l'ajoute via Paramètres → Connecteurs → Ajouter un connecteur personnalisé, et les deux outils LinkedIn apparaissent.

Ce qui ne traverse pas (encore) — les skills Claude Code (le format SKILL.md avec frontmatter qu'on voit dans Q1, Q2). Pour les rendre disponibles dans Cowork il faut soit soumettre le plugin au catalogue Anthropic, soit coller le corps du skill dans les instructions personnalisées d'un Project Cowork. La visite guidée elle-même est aussi Claude Code seulement (elle est faite de skills).

J'ai écrit une page parallèle qui reprend chacune des six questions avec le chemin Cowork détaillé — ce qui est pareil, ce qui change. Plus un guide de déploiement Cloudflare Workers (gratuit, 5 minutes) et une note sur OAuth pour la production.

Doc Cowork : docs/07-using-with-cowork.fr.md  ·  Déploiement : cloudflare/README.md

Si tu lances la visite avec le mode cowork, elle saute les sujets Claude-Code-seulement (bang-blocks, hooks, CLAUDE.md) et reformule Q4-Q5 autour du transport HTTP : 

/showcase-tour:tour cowork fr

5.Trois portes d'entrée pour explorer

Si tu veux creuser plus loin que la visite guidée, le plugin showcase-tour a deux autres skills : 

/showcase-tour:explain skills              # leçon focalisée · ~5 min
/showcase-tour:inspect plugins/linkedin-post   # marche-arrière du code

explain a un catalogue de 13 leçons (skills, plugins, marketplaces, MCP, hooks, etc.). inspect lit n'importe quel fichier du repo et te le commente ligne par ligne.

Si tu bloques quelque part ou si tu veux qu'on en jase en personne — c'est plus efficace en direct quand il y a des choix d'archi à faire. Bonne lecture.
